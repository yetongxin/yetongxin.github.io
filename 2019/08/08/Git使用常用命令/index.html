版本回退

HEAD指向的版本就是当前版本，因此，Git允许我们在版本的历史之间穿梭，使用命令git reset –hard commit_id
穿梭前，用git log可以查看提交历史，以便确定要回退到哪个版本。
要重返未来，用git reflog查看命令历史，以便确定要回到未来的哪个版本。

Git 撤销修改

checkout -- readme.txt

改乱了工作区的内容（已经add），丢弃工作区可以用：git reset HEAD <file>后checkout -- readme.txt

Git 删除

git rm删掉，并且git commit

Git 恢复删除

删错时候因为版本库里还有，所以可以很轻松地把误删的文件恢复到最新版本：
$ git checkout – filename
git checkout其实是用版本库里的版本替换工作区的版本，无论工作区是修改还是删除，都可以“一键还原”。

关联远程库

要关联一个远程库，使用命令
git remote add origin git@server-name:path/repo-name.git；
关联后，使用命令git push -u origin master第一次推送master分支的所有内容；
此后使用命令git push origin master推送最新修改；

分支

  • 创建分支：git checkout -b dev 创建分支并切换

  • 切换分支：git checkout <name>

  • 列出所有分支：git branch命令会列出所有本地分支，git branch -a列出所有远程与本地分支

  • 合并分支：git merge命令用于合并指定分支到当前分支

  • 删除分支：git branch -d dev

  • 把某个分支上的内容都拉取到本地 git pull origin dev

  • 拉取远程分支并创建本地分支：git checkout -b 本地分支名x origin/远程分支名x，此种方法建立的本地分支会和远程分支建立映射关系
    第二种方法是：git fetch origin 远程分支名x:本地分支名x，然后手动Checkout，此种方法建立的本地分支不会和远程分支建立映射关系。

    • 本地分支和远程分支建立映射关系的作用
      使用git pull或者git push时就不必每次都要指定从远程的哪个分支拉取合并和推送到远程的哪个分支了。
      git branch -vv查看映射关系

  • 手动建立本地分支和远程分支的映射关系：git branch -u origin/分支名(没用过)，或者git branch --set-upstream-to origin/分支名(用过)

  • 撤销本地分支与远程分支的映射关系：git branch --unset-upstream

  • 是否和远端的仓库建立连接：git remote -v

    实际开发的分支策略

    在工作的时候我们是在develop分支上干活，当有新功能的时候，从develop分支上checkout出一个feature分支进行开发，
    测试完成后提出merge request请求合并到develop分支上

在一开始接收老项目的时候，先git clone下来，这个时候应该就会在master分支上了，
接下来git checkout -b develop（本地分支） origin/develop（远程分支），此种方法建立的本地分支会和远程分支建立映射关系。
接下来基于develop分支下创建一个feat/featurename，
然后推送push到远程，因为现在远程还没有feat/featurename分支，所以需要set-upstream,第二次及以后就可以直接Push了：
git push --set-upstream origin feat/featurename

后续Push可以先pull一下，然后手动解决冲突后add commit push

git fetch

多人开发的时候，在自己写代码的时候肯定有别人已经提交代码到远程仓库上了，git fetch的作用就是将别人提交的这些记录拉到本地，没有别人的更新就不能checkout到对应的分支上咯

基本命令如下：
$ git fetch <远程主机名>
一般都是git fetch origin

如果只想拉某个确切的分支：
$ git fetch <远程主机名> <分支名>
比如取回远程的develop分支的更新，可以使用：$ git fetch origin develop

当拿到远程的更新后，就可以切换到这些已经更新了的分支上，下面这条命令很常用：
$ git checkout -b newBranch（本地分支名） origin/feature/featureName(远程分支名)
上面表示在origin/feature/featureName的基础上，创建一个新分支。

如果是多人开发同一个分支的话，还应该在这个newBranch检出一个自己的分支进行开发。在开发完成时拉取远程该分支的更新后，切换到该分支，merge自己写的分支（这一块是个人理解，有问题的话会修改）

git pull

前面的git fetch只是拉取远程的更新，没有进行合并，若要合并需要手动进行
而git pull命令的作用是，取回远程主机某个分支的更新，再与本地的指定分支合并。
基本命令如下：

如果省略本地分支名,如git pull origin origin/develop，就是远程分支是与当前分支合并,
相当于

Git会自动在本地分支与远程分支之间，建立一种追踪关系（tracking）。比如，在git clone的时候，所有本地分支默认与远程主机的同名分支，建立追踪关系，也就是说，本地的master分支自动”追踪”origin/master分支。也可以手动建立追踪关系:
git branch --set-upstream master origin/next
上面命令指定master分支追踪origin/next分支。

如果当前分支与远程分支存在追踪关系，git pull就可以省略远程分支名:$ git pull origin
如果当前分支只有一个追踪分支，连远程主机名都可以省略。$ git pull

git push

git push命令用于将本地分支的更新，推送到远程主机。它的格式与git pull命令比较像，push 本地 : 远程，pull 远程 : 本地，两者顺序相反
git push命令的格式如下：
$ git push <远程主机名> <本地分支名>:<远程分支名>
比如有用到git push origin HEAD:/orgin/feature/featureName，表示将当前分支（HEAD）推送到远程分支上

关于省略:

  1. 省略远程分支名，表示将本地分支推送与之存在”追踪关系”的远程分支（通常两者同名），如果该远程分支不存在，则会被新建。
    $ git push origin master将本地的master分支推送到origin主机的master分支。如果后者不存在，则会被新建。
  2. 省略本地分支名，表示删除指定的远程分支，因为这等同于推送一个空的本地分支到远程分支。
    git push orgin :origin/develop
  3. 如果当前分支与远程分支之间存在追踪关系，则本地分支和远程分支都可以省略: $ git push origin

git fetch和git pull区别：

git fetch <远程主机名> //这个命令将某个远程主机的更新全部取回本地,只是手动指定了要fetch的remote。在不指定分支时通常默认为master
git fetch <远程主机名> <分支名> //只想取回特定分支的更新,如取回master: git fetch origin master
git fetch是将远程主机的最新内容拉到本地，用户在检查了以后决定是否合并到工作本机分支中。
而git pull 则是将远程主机的最新内容拉下来后直接合并，即：git pull = git fetch + git merge，这样可能会产生冲突，需要手动解决。

git pull 的过程可以理解为：
git fetch origin master //从远程主机的master分支拉取最新内容
git merge FETCH_HEAD //将拉取下来的最新内容合并到当前所在的分支中
